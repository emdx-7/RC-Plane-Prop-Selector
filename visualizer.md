## Original goal (from the user)

Two separate tools:

1. **Candidate picker** (Part A вЂ” built first)
   - A long list of propellers, one **horizontal summary card / table row** each,
     showing all the prop's summary fields.
   - **Filter and sort by any field** (client side, instant).
   - **Selection model (up to 12 props):** check rows to select. Plots are NOT
     drawn per-row on click. Instead you build plots for the *selected set*:
     - Toolbar controls: **RPM-range slider**, **velocity slice slider**,
       **4 quantity toggles** (Thrust / Power / Efficiency / Torque),
       **simulated / empirical** enable toggle.
     - A **Build / Update** button pulls the sweep data for the selected props and
       (re)builds their cards, each with a **2D slice plot** (dependent vars vs RPM).
     - Sliders/toggles do NOT live-update the plots вЂ” they only take effect on the
       next Build press (same pattern as the demo's Rebuild button).
   - A **Deploy** button launches the in-depth visualizer (Part B) with the selected
     props. Works with 1вЂ“12 (does not require any fixed count). For now Deploy is a
     placeholder alert listing the selected names.
   - Keep the HTML clean, not sloppy.

### Data delivery (no web server; opens from file://)

87 MB of full sweeps is too big to embed. Instead `build_picker.py` writes a
**compact bundle** `picker_data.js` (~13 MB): for every prop, a slimmed sweep
(sim: rpm, V_mph, thrust, power, eff, torque; emp: rpm, J, CT, CP, eta) plus its
summary fields. `picker.html` loads Plotly + `picker_data.js` once, so filter/sort
and Build all work offline with no server. Build reads the selected props out of the
already-loaded bundle вЂ” no fetch.

2. **In-depth visualizer** (Part B вЂ” BUILT, generated by the Deploy button)
   - Modeled on `obs/demo.html`. A second "dep" checkbox column in the picker selects
     up to 4 foils; Deploy opens a self-contained new tab showing, per prop: three 3D
     scatters (Thrust/Power/Efficiency over RPMГ—speed), two 2D speed slices (0 & 40 mph),
     all on shared axes across the deployed props, with empirical (UIUC) points overlaid
     as hollow markers and a camera-sync Rebuild button. The generated tab loads
     `plotly.min.js` + `picker_data.js` from the picker's folder (absolute URLs), so it
     runs offline. All Part B code lives in `picker.html` (`deploy()`, `VISUALIZER_JS`,
     `VISUALIZER_CSS`) вЂ” no separate build step or file.

### `propeller/<name>.json` shape (what the picker reads)

```
{
  "meta": { name, diameter_in, pitch_in, blades, type[], apc_name, uiuc_name,
            has_simulated, has_empirical },
  "sources": {
    "simulated": {                      # APC вЂ” dimensional, or null
      rpm_min, rpm_max, v_max_mph, max_thrust_N, max_power_W,
      "sweep": [ { rpm, V_mph, J, Ct, Cp, eff, thrust_N, power_W,
                   torque_Nm, mach }, ... ]     # one row per (rpm, speed) point
    },
    "empirical": {                      # UIUC вЂ” coefficient-only, or null
      rpm_min, rpm_max, n_points, volumes[], geometry[],
      "sweep": [ { kind:"dynamic"|"static", rpm, J, CT, CP, eta }, ... ]
    }
  }
}
```

## Key decisions (locked in with the user)

- **Plot both sources**, with a simulated/empirical enable toggle in the toolbar.
- **Velocity slice via linear interpolation**, NEVER extrapolate. For each RPM row
  in the simulated grid, interpolate the quantity to the exact target speed; if the
  target speed is outside that row's measured speed range, that RPM contributes no
  point (no extrapolation).
- **Empirical is coefficient-only**, so we **convert it to dimensional units** to
  share the Thrust/Power/Torque axes. Standard prop-coefficient physics
  (ПЃ = 1.225 kg/mВі, D = diameter_in Г— 0.0254 m, n = rpm/60):
    - Thrust:  `T = CT В· ПЃ В· nВІ В· DвЃґ`         (N)
    - Power:   `P = CP В· ПЃ В· nВі В· DвЃµ`         (W)
    - Torque:  `Q = P / (2ПЂВ·n)`               (NВ·m)
    - Velocity from advance ratio: `V = J В· n В· D` (m/s), Г—2.23694 в†’ mph
  Efficiency uses `eta` directly. Because empirical points now have a velocity,
  the same velocity-slice logic applies to them.
- **Deploy button = placeholder alert** for now (lists selected prop names).
  Wire to the real Part B launch once Part B exists.

## Open issues / bugs

- **[FIXED] UNIT BUG вЂ” some UIUC props were in mm, labeled as inches.** Fixed in
  `data/UIUC/build_uiuc.py` (`mm_to_inch`, MM_CUTOFF_IN=40): any UIUC diameter > 40"
  is really mm (real props top out at 21", then a clean gap to 57mm+). Such props get
  dia & pitch converted mmв†’in, rounded to 0.5", and the size string prefixed with `~`
  (e.g. pl_100x80 в†’ pl_~4x3). 8 props converted. Rebuilt the whole DB
  (build_uiuc в†’ build_propeller в†’ build_picker). NOTE: rebuilding leaves stale old-name
  files behind вЂ” had to delete them (propeller/ and UIUC outputs/) before recounting.

- **[REVERTED] 3D surface method.** Briefly replaced the `mesh3d` sim surface with a
  true Plotly `surface` (rebuilt from the rectangular grid via `toGrid`) to smooth jagged
  edges, but on real data the `surface` version looked WORSE (more spiking), so we reverted
  to the original translucent `mesh3d` (Delaunay, opacity 0.45). `toGrid` was removed.
  Empirical stays as solid colored scatter markers on top. No points dropped either way.

## Notes as we go

- **Thrust reference line (both tools):** a typed thrust input (N) draws a thin
  translucent blue horizontal line at that thrust on the Thrust axis, applied on
  rebuild (Ctrl+B), only shown when Thrust is being plotted.
  - Visualizer: one per velocity slice (V1/V2), inside each slice's toolbar pill; the
    V slider + thrust input are linked by a shared slice identity color (V1 pill = teal
    #1BA39C, V2 pill = orange #E07A20) вЂ” no linking text.
  - Candidate picker: one thrust input next to the single velocity slider (blue-bordered
    `.tin`), applied on Build plots.
- **Speed-axis cap:** the shared 3D/2D speed (y) axis is capped at the max speed where
  thrust is still > 0 across deployed props (APC computes sweeps out past where thrust
  goes negative, ~250+ mph, which squashed the useful region). No data dropped вЂ” the
  negative-thrust tail sits beyond the visible axis. NOTE: props with very different
  pitch genuinely have very different speed envelopes (2" pitch ~110mph vs 8" ~280mph),
  so a mixed set still looks squashed on the SHARED axis вЂ” that's physical, not a bug.
  User chose to keep the shared axis (over per-prop speed axes).

- Deploy 3D plots: sim data is a **translucent `mesh3d`** (opacity 0.45, Delaunay),
  with empirical as **solid colored scatter markers** on top. No points dropped.
  (A `surface` variant was tried to remove spikes but looked worse вЂ” reverted.)
- **Filter box operators:** a column filter starting with `>`, `<`, `=`, `>=`, or `<=`
  is parsed as a numeric threshold on that column (else it's a substring match). Logic
  in `passesFilters`. Documented in the yellow filter-help note at the top of the page.
- **`~` name prefix** marks an approx-inches conversion of a mm-sized UIUC prop.

- Velocity slider is in **m/s** (data stored in mph; converted at slice time via
  MPH_TO_MS = 0.44704). Vmax table column stays in mph.
- **Layout rework:** table takes the left **3/4** (`.tablewrap`), plots live in the
  right **1/4** (`.plotcol`, position:relative). On Build, each selected prop's row
  is grown to `PLOT_H` px (dead space, pushes later rows down) and its plot is pinned
  absolute in the plot column at that row's vertical offset вЂ” so each plot sits next
  to its own summary row. A table re-render (filter/sort) clears all built plots.
- **Y-axis scaling replicates the demo:** for each quantity, every card uses
  `[0, sharedMax]` where sharedMax = max of that quantity across ALL built props
  (within the current RPM range / velocity slice). So thrust max is identical on
  every card, power/torque max identical. Efficiency is fixed [0,1].
- **Axes shown:** only thrust (left) and Power-OR-Torque (right, one axis) get a
  visible axis + legend. Efficiency has NO axis/ticks/legend вЂ” it's drawn on a
  hidden 0-1 axis with just a green dotted 100% reference line at 1.0 (demo style).
- **Power/Torque switch:** power and torque are the same quantity in different
  units (torque = power/(2*pi*n)), and every prop that has one has the other (data
  checked, 0 mismatches). So the toolbar shows one Power/Torque toggle plus a small
  switch button (`ptSwitch`, в‡„) that flips which unit the shared right axis shows.
  Power is the default. Defaults: thrust + power + efficiency on; empirical on.
